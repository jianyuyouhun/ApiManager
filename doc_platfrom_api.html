接口安全设置

  • 请求有效性

为了防止同一个请求无数次调用，每个请求有效时间为五分钟；过期请求失效不再处理改请求；为了您的请求的数据安全和有效“timestamp”请使用当前时间戳。

  • 数据安全性

为防止数据在传输过程中被截取和篡改，确保数据安全；接口请求参数和通用必传参数除“sign”需参与签名（签名详见附录一）。

  • 请求方式

  • 必传参数使用header头传输

  • 接口业务参数按接口定义的GET或POST方式传输

通知机制

一账户、一地址、多通知、按授权

一账号

每个接入平台分配一个账户（app_id）和密钥(app_key)

一地址

每个账号配置一个通知地址，当前用户定制的通知类型发生改变时往账号配置的通知地址通知变动消息；

多通知

现阶段通知内容类型有：医院信息、科室信息、医生信息、医生所属科室 对应t_sys_platform_user_cfg.notify_type

EG ######

//接收通知类型:医院信息和医生信息 1010000000 //接收通知类型：医院信息、科室信息、医生信息、医生所属科室 1111000000

按授权

每个平台只接收自己平台相关的信息，如预约挂号我只接收与挂号平台有关的数据；若需要接收多个平台的数据，“,”隔开配置多个授权即可。

EG ######

//需要接收挂号和互联网医院相关数据 is_auth_reg,is_auth_hlwyy

通知机制
  • 平台会把结果通知各个授权账号，各个子平台需要接收处理，并返回应答；应答数据结构： { "code":0,//0-为处理成功 "message":"" }
  • 如果平台收到子平台的应答不是成功或超时，平台认为通知失败，平台会通过一定的策略定期重新发起通知，尽可能提高通知的成功率，但平台不保证通知最终能成功。 （通知频率为15/15/30/180/1800/1800/1800/1800/3600，单位：秒）
  • 注意：同样的通知可能会多次发送给商户系统。商户系统必须能够正确处理重复的通知。
  • 注意：同样的通知可能会多次发送给子平台系统。子平台系统必须能够正确处理重复的通知。
  • 特别提醒：子平台系统对于通知的内容一定要做签名验证，防止数据泄漏导致出现“假通知”。

API请求方式

  • 必填参数使用Headers传参

app-id:10000

  • 接口业务参数

业务参数按接口定义使用POST或者GET请求

  • 签名

  • 请求的必填参数与接口定义的POST或者GET请求参数，需要根据参数名=参数值的格式，按首字母（ASCII值大小）排序，若遇到相同字符，则判断第二个字符，以此类推，待签名字符串需 要以“参数名 1=参数值 1&参数名 2=参数值 2&....&参 数名 N=参数值 N”的规则进行拼接。

  • 在对请求的参数做签名时，这些参数必须来源于请求参 数列表，并且除去列表中的参数 sign。
  • 在对请求的参数做签名时，对于请求参数列表中那些可 空的参数，如果选择使用它们，那么这些参数的参数值 必须不能为空或空值。
  • 待签名数据应该是参数原始值而不是 url encoding 之 后的值，例如:调用某接口需要对请求参数 email 进行 数字签名，那么待签名数据应该是 email=test@msn.com，而不是 email=test%40msn.com。
  • 在字符串后拼接“&key=密钥”
  • 将最后拼接字符串使用MD5加密
